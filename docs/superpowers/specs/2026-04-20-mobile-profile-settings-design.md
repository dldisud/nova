**제외 범위**: 결제/코인, 언어 설정, 고객센터, 아바타 업로드, 계정 삭제, 서재 액션, `mob/` 웹 목업  
**원칙**: 이미 연결된 Supabase 세션/프로필 읽기 흐름을 유지하면서, 설정 쓰기와 안전한 로그아웃만 실제 기능으로 완성한다

## 목표

이번 단계의 목표는 `ProfileScreen`을 읽기 전용 화면에서 실제 계정 설정 화면으로 확장하는 것이다.

대상 기능:

1. 현재 닉네임을 불러와 수정하고 저장
2. 마케팅 알림 수신 여부를 조회하고 저장
3. 로그아웃 실패를 사용자에게 명확히 알려주는 안전한 로그아웃 처리

이번 단계는 “프로필 기본 설정을 실제 Supabase에 저장할 수 있는 상태”를 만드는 것이 목표이며, 결제/코인과 같은 미구현 백엔드 계약은 포함하지 않는다.

---

## 제안한 접근과 선택

### 접근안 A. `ProfileScreen` 내부 인라인 편집

- 현재 `ProfileScreen` 안에 설정 폼을 추가하고 저장 버튼을 둔다
- 읽기와 쓰기를 같은 사용자 흐름 안에서 끝낸다
- 장점: 화면 이동이 없고, 기존 구조를 가장 적게 흔든다
- 단점: 화면 컴포넌트가 조금 더 커진다

### 접근안 B. 설정 전용 화면 분리

- `ProfileScreen`은 요약 카드만 보여주고, 별도 설정 화면으로 이동한다
- 장점: 화면 책임이 더 깔끔하다
- 단점: 이번 범위에는 라우트와 UI가 불필요하게 늘어난다

### 접근안 C. 설정 전용 repository를 별도로 분리

- 읽기용 `accountRepository`와 별도로 쓰기 전용 설정 repository를 새로 둔다
- 장점: 장기적으로는 관심사가 더 또렷하다
- 단점: 이번 1차 범위에서는 추상화 비용이 더 크다

### 선택

**접근안 A. `ProfileScreen` 내부 인라인 편집**을 채택한다.

이유:

- 현재 `ProfileScreen`은 이미 `profiles`와 세션 데이터를 읽고 있으므로, 같은 맥락에서 설정 저장까지 연결하는 것이 가장 자연스럽다
- 결제/코인처럼 막혀 있는 스키마가 아니라 `profiles.display_name`, `profiles.marketing_opt_in`은 이미 존재한다
- 다음 단계에서 설정 범위가 커지면 그때 전용 화면이나 전용 repository로 분리해도 늦지 않다

## 아키텍처

### 읽기 경계 유지

기존 `createAccountRepository().getProfileData()`는 계속 유지한다.

- 프로필 표시 이름
- 이메일
- 작가 여부
- 활동 통계

는 지금처럼 읽어 온다.

### 쓰기 경계 추가

`inkroad-app/src/mobile/reader/accountRepository.ts`에 설정 저장용 메서드를 추가한다.

예상 책임:

- `display_name` 검증 후 `profiles` 업데이트
- `marketing_opt_in` 업데이트
- 저장 완료 후 최신 profile 데이터를 화면에 반영할 수 있는 결과 반환

화면은 Supabase client를 직접 업데이트하지 않고 repository 메서드만 호출한다.

### 로그아웃 처리

로그아웃은 지금처럼 `supabase.auth.signOut()`를 사용한다.

다만 이번 단계에서 다음을 보강한다.

- 로그아웃 진행 중 버튼 중복 탭 방지
- 실패 시 오류 메시지 표시
- 성공 시 `/auth`로 이동

## 데이터 계약

이번 단계에서 실제로 읽고 쓰는 계정 필드는 다음으로 고정한다.

### 읽기

- `profiles.id`
- `profiles.display_name`
- `profiles.role`
- `profiles.marketing_opt_in`
- 세션의 `user.email`
- `authors.user_id` 존재 여부를 통한 `isCreator`

### 쓰기

- `profiles.display_name`
- `profiles.marketing_opt_in`

### 검증 규칙

- 닉네임은 공백 제거 후 비어 있으면 저장하지 않는다
- 닉네임은 지나치게 긴 값을 막기 위해 길이 제한을 둔다
- 마케팅 알림은 boolean 하나만 저장한다

이번 단계에서는 다음 필드는 저장 대상에서 제외한다.

- `avatar_url`
- `role`
- 저자 `pen_name`
- 언어 설정
- 세부 알림 카테고리

## 화면 동작

### 로그인 상태

로그인된 사용자는 기존 프로필 카드 아래에 설정 섹션을 본다.

설정 섹션 구성:

- 닉네임 입력 필드
- 마케팅 알림 수신 토글
- 저장 버튼
- 저장 상태 메시지

초기값은 repository가 읽어 온 현재 `profiles` 값으로 채운다.

### 비로그인 상태

지금과 동일하게 로그인 유도 화면만 보여준다.

설정 폼은 렌더링하지 않는다.

### 저장 흐름

1. 사용자가 닉네임/마케팅 알림 값을 수정한다
2. 저장 버튼을 누르면 버튼 비활성화와 함께 저장 중 상태를 표시한다
3. 저장 성공 시:
   - 프로필 카드의 이름이 즉시 갱신된다
   - 성공 메시지를 표시한다
4. 저장 실패 시:
   - 기존 값은 화면에 유지한다
   - 실패 메시지를 표시한다

### 로그아웃 흐름

1. 사용자가 로그아웃 버튼을 누른다
2. 로그아웃 요청 중에는 버튼을 다시 누를 수 없게 한다
3. 성공 시 `/auth`로 이동한다
4. 실패 시 현재 화면에 머물고 오류 메시지를 보여 준다

## 오류 처리

이번 단계의 오류 처리는 세 가지로 나눈다.

1. 프로필 조회 실패
2. 설정 저장 실패
3. 로그아웃 실패

표시 원칙:

- 조회 실패는 기존 화면의 안내 문구 영역에 표시한다
- 저장 실패는 설정 폼 바로 아래 상태 메시지로 표시한다
- 로그아웃 실패는 사용자 행동과 직접 연결되므로 명확한 에러 메시지를 보여 준다

에러가 나더라도 세션이 살아 있다면 화면 전체를 비우지 않고, 마지막으로 로드된 값을 계속 보여 준다.

## 테스트 전략

이번 단계는 테스트를 먼저 작성하고 구현한다.

필수 테스트:

1. `ProfileScreen`이 repository 데이터로 닉네임과 마케팅 알림 초기값을 채우는지
2. 저장 버튼이 `updateProfileSettings()`를 올바른 값으로 호출하는지
3. 저장 성공 후 화면 이름이 갱신되는지
4. 저장 실패 시 에러 메시지가 보이는지
5. 로그아웃 성공 시 `supabase.auth.signOut()` 후 `/auth`로 이동하는지
6. 로그아웃 실패 시 라우팅하지 않고 실패 메시지를 보여 주는지

기존 `ProfileScreen` 테스트는 유지하면서, 설정 저장/실패 경로를 검증하는 케이스를 추가한다.

## 구현 순서

1. `ProfileScreen` 설정 저장/로그아웃 실패 동작을 설명하는 테스트를 먼저 추가한다
2. `accountRepository`에 `marketing_opt_in` 읽기와 `updateProfileSettings()`를 추가한다
3. `ProfileScreen`에 설정 폼, 저장 상태, 로그아웃 오류 처리를 연결한다
4. 테스트를 다시 돌려 green 상태를 확인한다
