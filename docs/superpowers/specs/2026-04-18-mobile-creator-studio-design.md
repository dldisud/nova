# 모바일 작가 스튜디오 설계 문서

**날짜**: 2026-04-18  
**범위**: `inkroad-app/` Expo 앱 1차 구현 + `mob/` 모바일 웹으로 이어질 공통 정보구조 정의  
**원칙**: 글쓰기 중심, 작가가 길을 잃지 않게, 문법보다 버튼과 흐름이 먼저 보이게

---

## 목표

InkRoad 모바일 앱에 작가 전용 공간을 추가해, 작가가 모바일에서도 다음 흐름을 실제로 수행할 수 있게 만든다.

1. 내 작품 목록 확인  
2. 작품 선택  
3. 회차 목록 확인  
4. 새 회차 작성  
5. 본문 미리보기 및 발행

이 작업은 단순 관리 화면 추가가 아니라, “작가가 앱 안에서 실제로 글을 쓰는 흐름”을 여는 것이 목표다.  
또한 이후 `mob/` 기반 모바일 웹 작가 페이지를 만들 때 같은 정보구조와 화면 경계를 재사용할 수 있어야 한다.

---

## 제품 방향

### 왜 모바일 작가 공간이 필요한가

현재 앱은 독자 흐름은 존재하지만, 작가 흐름은 없다.  
그래서 작가는 모바일에서 자기 작품을 보거나 회차를 쓰는 일 자체가 막혀 있다.

이 상태를 바꾸기 위해, 모바일에서도 “작가용 홈”이 따로 보여야 한다.

### 핵심 원칙

1. **읽기 앱처럼 복잡하지 않게**  
   작가 공간은 독자 화면의 연장선이 아니라, “작업 공간”처럼 느껴져야 한다.

2. **글쓰기 우선**  
   통계, 설정, 부가 관리보다 `새 회차 쓰기`와 `최근 작업 이어쓰기`가 가장 앞에 와야 한다.

3. **문법보다 버튼**  
   연출형 서식은 직접 외우는 방식이 아니라 `버튼 -> 삽입 -> 즉시 미리보기`로 접근해야 한다.

4. **모바일에 맞게 단순화**  
   PC의 모든 운영 기능을 그대로 축소 복사하지 않는다.  
   모바일에서는 `작품`, `회차`, `본문`, `발행` 순서를 더 강하게 드러낸다.

---

## 제안한 접근과 선택

### 접근안 A. MY 내부 진입형

- `MY` 안에서 `작가 모드`로 들어감
- 장점: 기존 탭 구조를 덜 흔든다
- 단점: 작가 기능이 숨어 보인다

### 접근안 B. 하단 탭 작가 분리형

- 하단 탭에 `작가`를 별도로 추가
- 안에 작품 관리, 회차 관리, 작성 화면을 둔다
- 장점: 작가가 바로 진입 가능하고 길을 잃지 않는다
- 단점: 일반 독자 기준 탭이 하나 늘어난다

### 접근안 C. 새 회차 쓰기 우선 단축형

- 작품 관리보다 `새 회차 쓰기`를 가장 먼저 열어주는 구조
- 장점: 즉시 글쓰기에 강하다
- 단점: 관리 체계가 약하고 작품 간 이동이 불편하다

### 선택

**접근안 B. 하단 탭 작가 분리형**을 채택한다.

이유:

- 사용자 요청이 “모바일에서도 글쓰기 기능이 있어야 한다”는 점에 직접 맞는다
- 숨어 있는 진입보다 전용 탭이 훨씬 명확하다
- 이후 모바일 웹 작가 페이지를 만들 때도 같은 구조를 옮기기 쉽다

## 정보 구조

모바일 작가 공간의 1차 정보 구조는 다음과 같다.

### 1. 작가 홈

작가가 앱에서 가장 먼저 보는 화면이다.

포함 요소:

- 상단 요약
  - 내 작품 수
  - 최근 수정한 시각
- 큰 CTA
  - `새 회차 쓰기`
  - `최근 작업 이어쓰기`
- 내 작품 목록
  - 표지
  - 제목
  - 상태
  - 회차 수
  - 최근 수정일

목적:

- “무엇을 써야 하는지”를 바로 보게 하기
- 관리보다 실행을 앞세우기

### 2. 작품 관리 상세

특정 작품 하나를 눌렀을 때 들어가는 화면이다.

포함 요소:

- 작품 표지/제목/한 줄 정보
- 작품 상태
- 회차 수
- `새 회차 작성` 버튼
- 회차 목록
  - 제목
  - 회차 번호
  - 무료/유료
  - 최근 수정 상태

목적:

- 작품 단위로 글쓰기 흐름을 정리
- 어떤 작품의 어떤 회차를 다루고 있는지 분명하게 보여주기

### 3. 회차 작성 화면

실제 본문을 쓰는 핵심 화면이다.

포함 요소:

- 작품 선택
- 회차 제목
- 공개 방식(무료/유료)
- 가격 입력(유료일 때만)
- 본문 입력
- 연출 서식 도우미 진입 또는 인라인 버튼
- 실시간 미리보기
- 저장/발행 버튼

목적:

- 모바일에서도 실제 회차를 완성할 수 있게 하기
- 연출 서식 기능을 “작가가 쓸 수 있는 수준”으로 낮추기

---

## 화면 경계

### Expo 앱 기준

예상 화면 경계는 다음과 같다.

- `작가 탭 홈`
- `작품 관리 상세`
- `회차 작성`
- `연출 서식 도우미`

가능하면 기존 앱 구조를 깨지 않도록 Expo Router 기준으로 새 route를 추가한다.

### 모바일 웹 기준

`mob/`에는 아직 모바일 작가 페이지가 없으므로, 이후 같은 구조를 HTML 페이지로 옮긴다.

모바일 웹 1차 목표:

- 앱과 같은 정보 구조를 유지
- 시각 톤은 웹 스타일을 따르되, 흐름은 앱과 동일하게 맞춤

즉 모바일 웹은 별도 설계를 새로 만드는 것이 아니라, **앱에서 먼저 검증한 구조를 재사용**하는 방향으로 간다.

---

## 데이터 구조

1차 구현은 목업 데이터 기반으로 시작하되, 나중에 Supabase로 바꾸기 쉬워야 한다.

### 필요한 데이터 단위

- `AuthorWorkSummary`
  - 작품 id
  - 제목
  - 표지
  - 상태
  - 회차 수
  - 최근 수정일

- `AuthorEpisodeSummary`
  - 회차 id
  - 번호
  - 제목
  - 무료/유료
  - 가격
  - 마지막 수정일

- `EpisodeDraft`
  - 작품 id
  - 회차 id 또는 신규 여부
  - 제목
  - 공개 방식
  - 가격
  - 본문

### 저장소 경계

기존 모바일 코드의 저장소/목업 패턴을 유지하면서 아래 성격의 인터페이스를 둔다.

- `AuthorRepository`
  - 내 작품 목록
  - 작품별 회차 목록
  - 회차 초안 로드
  - 회차 저장/발행

1차는 `MockAuthorRepository`로 충분하다.  
나중에 Supabase 저장소로 갈아끼우더라도 화면 코드가 크게 흔들리지 않아야 한다.

## 연출 서식 작성 경험

이 부분은 매우 중요하다.  
기존의 “문법 직접 입력” 방식은 기능으로는 가능해도, 코드를 모르는 작가에게는 장벽이 높다.

### 원칙

- 일반 본문은 그냥 쓴다
- 특수 연출만 버튼으로 넣는다
- 넣은 뒤 바로 아래 미리보기에서 확인한다

### 1차 지원 요소

- 루비
- 상태창
- 시스템 메시지
- 경고문
- 채팅창
- 회상
- 기사문
- 기술명 블록

### UX 방식

- 본문 입력 아래에 `연출 넣기` 버튼 모음
- 버튼을 누르면 예시 블록이 자동 삽입
- 라이브 미리보기로 즉시 확인

즉 작가는 “문법을 쓴다”가 아니라 **“버튼으로 연출을 넣는다”**고 느껴야 한다.

## 시각 디자인 방향

### 작가 공간의 시각 톤

독자 화면과 완전히 분리하지는 않지만, 목적이 다르므로 조금 더 작업실 같은 톤으로 바꾼다.

- 어두운 배경 유지
- 금색 포인트 유지
- 카드와 입력창은 더 명확한 작업용 레이아웃
- 큰 CTA는 금색 또는 밝은 강조색

### 글쓰기 화면

- 본문 입력이 가장 커야 한다
- 작품 정보/공개 방식은 상단에서 짧게 정리
- 미리보기는 접거나 아래쪽 카드로 배치
- 서식 도우미는 시각적으로 친절해야 하며 “기능 목록”처럼 보여야 한다

### 모바일 웹과의 관계

모바일 웹도 같은 구조를 따라가되, HTML/CSS 한계상 앱보다 조금 더 단순한 카드 중심 레이아웃으로 간다.

## 에러 처리와 상태 처리

### 작성 중 상태

- 초안 저장 중
- 저장 완료
- 발행 준비 완료
- 발행 실패

상태는 문장으로 분명하게 보여준다.

예:

- `저장 중입니다`
- `방금 저장됐어요`
- `필수 항목을 먼저 입력해 주세요`
- `네트워크 오류로 저장하지 못했습니다`

### 안전장치

- 제목 없는 발행 방지
- 본문 없는 발행 방지
- 유료 회차인데 가격 없는 상태 방지
- 저장 실패 시 임시 초안 유지

## 테스트 기준

### 수동 확인

1. 작가 탭이 보이는가
2. 내 작품 목록이 보이는가
3. 작품 하나를 눌렀을 때 회차 목록이 열리는가
4. 새 회차 작성 화면으로 들어갈 수 있는가
5. 제목/본문 입력이 되는가
6. 연출 버튼을 눌렀을 때 본문에 블록이 들어가는가
7. 미리보기에서 실제 연출이 보이는가
8. 저장/발행 버튼 상태가 입력값에 따라 바뀌는가

### 앱/웹 공통 체크

- 구조가 지나치게 다르지 않은가
- 작가가 어디서 글을 써야 하는지 헷갈리지 않는가
- 모바일 화면에서 입력 영역이 너무 답답하지 않은가

## 범위 정리

### 이번 1차 범위

- Expo 앱에 작가 탭 추가
- 내 작품 목록
- 작품별 회차 목록
- 회차 작성 화면
- 연출 서식 도우미 연결
- 목업 데이터 기반 저장 흐름

### 이번 1차 범위 밖

- 완전한 Supabase 저장 연동
- 모바일 웹의 모든 페이지 구현 완료
- 통계 화면 모바일 완전 대응
- 관리자 기능 모바일 대응

단, 모바일 웹은 이후 바로 옮길 수 있도록 정보 구조를 이 문서 기준으로 고정한다.

## 구현 순서 제안

1. 작가 탭과 라우트 추가  
2. 작가 홈 화면 추가  
3. 작품 상세 관리 화면 추가  
4. 회차 작성 화면 추가  
5. 연출 서식 도우미 연결  
6. 목업 저장/발행 상태 처리  
7. 모바일 웹으로 동일 구조 이식

## 결정 요약

- 모바일 작가 기능은 `하단 탭 작가 분리형`으로 간다
- 1차는 `작품 관리 + 회차 목록 + 회차 작성`까지 한 번에 넣는다
- 연출 서식은 문법 직접 입력보다 `버튼 삽입 + 즉시 미리보기`를 우선한다
- 앱에서 먼저 구조를 검증하고, 이후 모바일 웹은 같은 구조를 재사용한다
