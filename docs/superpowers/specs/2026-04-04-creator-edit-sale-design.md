# 잉크로드 크리에이터 운영 확장 — 작품 수정 / 회차 수정 / 할인 등록

## 개요

현재 크리에이터 흐름은 `새 작품 등록`, `새 회차 발행`, `내 작품 보기`까지만 된다. 즉, 한 번 올린 뒤 다시 고치거나, 세일을 걸어서 다시 노출시키는 운영 동선이 빠져 있다.

이번 작업의 목적은 **기존 크리에이터 페이지를 재사용하면서, 실제 운영에 필요한 최소 기능을 붙이는 것**이다.

핵심 방향은 세 가지다.
- 작품 수정: 기존 `novel_upload_pc.html`을 재사용한다.
- 회차 수정: 기존 `episode_upload_pc.html`을 재사용한다.
- 할인 등록: `creator_dashboard_pc.html`에서 작품별 단일 할인만 바로 등록한다.

이 설계는 새 관리자 시스템을 크게 만드는 대신, 지금 있는 화면을 그대로 살리면서 `만들기 -> 수정하기 -> 할인 걸기` 흐름을 완성하는 데 초점을 둔다.

---

## 왜 이 방식으로 가는가

비슷한 기능을 새 페이지로 전부 분리하면 구조는 깔끔해 보이지만, 지금 단계에서는 파일 수가 급격히 늘고 같은 폼을 두 번 관리하게 된다. 반대로 업로드 화면을 그대로 재사용하면 다음 장점이 있다.

- 기존 UI와 학습 비용을 그대로 유지할 수 있다.
- 이미 구현된 로그인, 태그, 표지 업로드, 에디터, 상태 표시를 계속 활용할 수 있다.
- 수정 기능이 추가되어도 사용자가 새 화면을 다시 배울 필요가 없다.
- 구현 범위를 작게 잡을 수 있어 실제 동작까지 빨리 갈 수 있다.

따라서 이번 범위에서는 **새 파일을 많이 만들지 않고, 기존 업로드 화면에 `edit` 모드를 추가하는 방향**을 선택한다.

## 이번 범위

### 포함

- `novel_upload_pc.html?edit=<novel_id>` 지원
- `episode_upload_pc.html?edit=<episode_id>` 지원
- 크리에이터 대시보드에 `작품 수정`, `회차 수정`, `할인 등록` 진입점 추가
- 할인 등록 시 `novels` 할인 필드와 `events`, `event_items`를 함께 갱신
- 수정/등록은 로그인한 계정이 소유한 작품에서만 가능

### 제외

- 작품 삭제
- 회차 삭제
- 다중 작품 묶음 이벤트 편집기
- 관리자 전용 전체 운영 콘솔
- 복잡한 할인 히스토리/통계 화면

이번 목표는 “운영 가능한 최소 기능”이다. 삭제나 대규모 이벤트 관리는 다음 단계로 미룬다.

## 사용자 흐름

### 1. 작품 수정

1. 작가가 `creator_dashboard_pc.html`에서 `작품 수정` 버튼을 누른다.
2. `novel_upload_pc.html?edit=<novel_id>`로 이동한다.
3. 페이지는 현재 로그인 계정이 그 작품의 소유자인지 확인한다.
4. 소유 작품이면 기존 제목, 소개, 태그, 연령등급, 번역 여부, 원산지, 표지를 불러온다.
5. 사용자는 값을 수정하고 `저장하기`를 누른다.
6. 저장이 끝나면 작품 상세나 대시보드로 복귀할 수 있다.

### 2. 회차 수정

1. 작가가 `creator_dashboard_pc.html`에서 `최근 회차 수정` 버튼을 누른다.
2. `episode_upload_pc.html?edit=<episode_id>`로 이동한다.
3. 페이지는 해당 회차가 로그인한 계정의 작품에 속하는지 확인한다.
4. 회차 제목, 본문, 무료/유료, 가격을 기존 값으로 채운다.
5. 사용자가 수정 후 저장하면 같은 회차가 업데이트된다.
6. 저장 후 뷰어나 상세 페이지로 이동한다.

### 3. 할인 등록

1. 작가가 `creator_dashboard_pc.html`에서 작품 카드의 `할인 등록` 버튼을 누른다.
2. 같은 페이지 안에 간단한 할인 패널이 열린다.
3. 할인율, 시작일, 종료일을 입력한다. 기존 활성 할인이 있으면 그 값을 먼저 채워 보여준다.
4. 저장하면 작품의 할인 필드가 갱신되고, 이벤트 테이블에도 단일 작품용 이벤트가 연결된다.
5. 저장 후 홈 세일 배너, 탐색 정렬, 상세 가격 박스에 자동 반영된다.

## 화면 설계

### 1. 작품 업로드 페이지의 수정 모드

대상: `/mnt/d/nova/novel_upload_pc.html`, `/mnt/d/nova/assets/upload-studio.js`

### 모드 전환 규칙
- 쿼리에 `edit`가 없으면 기존처럼 새 작품 등록 모드
- `?edit=<novel_id>`가 있으면 작품 수정 모드

### 수정 모드에서 달라지는 점
- 페이지 제목: `새 작품 등록` -> `작품 수정`
- 보조 문구: `첫 회차와 함께 작품을 등록합니다` 계열 문구 -> `작품 기본 정보를 수정합니다`
- 첫 회차 입력 영역은 수정 모드에서 숨긴다
  - 이유: 작품 정보 수정과 첫 회차 생성은 성격이 다르다
- 버튼 문구: `발행하기` -> `저장하기`
- 표지를 새로 선택하지 않으면 기존 표지를 유지한다

### 불러올 데이터
- `novels`: 제목, 짧은 소개, 설명, 연령등급, 번역 여부, 원산지, 표지 URL
- `novel_tags` + `tags`: 현재 작품 태그

### 저장 규칙
- 제목, 소개, 태그, 연령등급, 번역 여부, 원산지, 표지 URL만 수정한다
- 회차나 판매 기록은 여기서 건드리지 않는다
- 기존 표지를 유지할 수 있어야 한다
- 새 표지를 올리면 새 URL로 교체한다

### 접근 제한
- 로그인 필수
- 해당 작품의 `author_id`가 현재 사용자 계정의 `authors.user_id`와 연결된 경우만 허용
- 소유자가 아니면 폼을 숨기고 오류 안내만 보여준다

---

### 2. 회차 업로드 페이지의 수정 모드

대상: `/mnt/d/nova/episode_upload_pc.html`, `/mnt/d/nova/assets/episode-upload.js`

### 모드 전환 규칙
- 쿼리에 `edit`가 없으면 기존처럼 새 회차 발행 모드
- `?edit=<episode_id>`가 있으면 회차 수정 모드

### 수정 모드에서 달라지는 점
- 페이지 제목: `회차 추가` -> `회차 수정`
- 선택 작품 드롭다운은 잠그거나 읽기 전용으로 처리한다
  - 이유: 수정 중에 회차를 다른 작품으로 옮기면 데이터 경계가 흐려진다
- 회차 번호는 표시만 하고 수정 대상에서 제외한다
- 버튼 문구: `회차 발행` -> `저장하기`
- 저장 후에는 새 회차가 아닌 기존 회차 뷰어로 이동한다

### 불러올 데이터
- `episodes`: 제목, 접근 유형, 가격, 회차 번호
- `episode_contents`: 본문
- 소속 `novels`: 제목, 표지, 태그, 상태

### 저장 규칙
- 제목, 본문, 무료/유료, 가격만 수정한다
- 회차 번호는 유지한다
- 무료로 바꾸면 가격은 자동으로 0 처리
- 유료로 바꾸면 가격 입력이 필요하다

### 자동저장/스냅샷과의 관계
- 기존 자동저장 로직은 유지한다
- 수정 모드에서는 현재 회차 ID가 명확하므로 스냅샷 기능이 실제로 의미 있게 동작한다
- 저장 전후 상태 문구는 `초안 저장`, `수정 저장 완료` 식으로 더 명확하게 나눈다

### 접근 제한
- 로그인 필수
- 로그인 사용자가 소유한 작품의 회차만 수정 가능
- 권한이 없으면 폼 대신 안내 카드 표시

### 3. 크리에이터 대시보드의 운영 패널

대상: `/mnt/d/nova/creator_dashboard_pc.html`, `/mnt/d/nova/assets/creator-dashboard.js`

### 목록 카드에 추가할 액션
- `작품 수정`
- `새 회차`
- `최근 회차 수정`
- `할인 등록`
- `상세 보기`

지금 있는 `새 회차`, `읽기 확인`은 유지하되, 운영 액션을 앞쪽으로 재배치한다.

### 대표 작품 영역에 추가할 액션
- `작품 수정`
- `할인 등록`
- `상세 보기`

### 할인 등록 UI
이번 범위에서는 모달보다 **인라인 패널 또는 슬라이드 패널**이 적합하다.

필드:
- 작품명(읽기 전용)
- 기본 편당 가격 안내: `편당 300원`
- 할인율: 5 / 10 / 15 / 20 / 30 / 40 / 50 중 선택
- 할인 시작일시
- 할인 종료일시
- 미리보기: `편당 300원 -> 편당 210원`
- 저장 버튼

### 단순화 원칙
- 할인은 작품별 단일 할인만 지원
- 한 작품에 같은 시점의 활성 할인은 하나만 둔다
- 복수 이벤트 동시 참여는 이번 범위에서 다루지 않는다

## 데이터 / 권한 설계

현재 스키마는 공개 조회와 독자 데이터 관리 중심이며, 작가가 `novels`, `episodes`, `events`, `event_items`를 직접 수정하는 정책은 없다. 따라서 클라이언트에서 테이블을 직접 업데이트하기보다, **소유권 검사를 내부에서 하는 RPC 함수**를 추가하는 방향이 안전하다.

### 추가할 RPC 함수

#### 1. `update_novel_for_author`
역할:
- 현재 로그인 사용자가 해당 작품의 소유자인지 확인
- 작품 기본 정보를 업데이트
- 필요하면 새 표지 URL로 교체
- 태그 연결을 재구성

입력 예시:
- `p_novel_id`
- `p_title`
- `p_short_description`
- `p_description`
- `p_age_rating`
- `p_is_translation`
- `p_origin_country`
- `p_cover_url`
- `p_tags`

반환:
- `novel_id`
- `novel_slug`
- `updated_at`

#### 2. `update_episode_for_author`
역할:
- 현재 로그인 사용자가 해당 회차의 소유자인지 확인
- 회차 메타와 본문을 업데이트
- 무료/유료 전환 시 가격 규칙을 강제

입력 예시:
- `p_episode_id`
- `p_title`
- `p_body`
- `p_access_type`
- `p_price`

반환:
- `episode_id`
- `novel_slug`
- `episode_number`
- `updated_at`

#### 3. `upsert_novel_sale_for_author`
역할:
- 현재 로그인 사용자가 해당 작품의 소유자인지 확인
- 할인율로 실제 할인 가격을 계산
- `novels.bundle_sale_price`, `sale_starts_at`, `sale_ends_at` 갱신
- 단일 작품용 이벤트를 `events`, `event_items`에 생성 또는 갱신
- 시작/종료 시각을 기준으로 이벤트 상태를 `scheduled` 또는 `active`로 결정

입력 예시:
- `p_novel_id`
- `p_discount_percent`
- `p_starts_at`
- `p_ends_at`

반환:
- `novel_id`
- `discount_percent`
- `sale_price`
- `event_id`
- `event_slug`

### 할인 저장 규칙
할인 등록은 두 군데를 함께 맞춰야 한다.

1. `novels`
- `bundle_list_price`는 기본값 유지
- `bundle_sale_price`에 할인 적용값 저장
- `sale_starts_at`, `sale_ends_at` 저장

2. `events`, `event_items`
- 이벤트 타입은 이번 범위에서 `bundle_sale`로 고정한다
- 한 작품 전용 이벤트를 만든다
- `event_items.discount_percent`, `event_items.sale_price`도 함께 저장

이렇게 해야 현재 홈/탐색/상세가 읽는 할인 정보와 세일 배너 구성이 동시에 맞는다.

### 왜 직접 UPDATE 정책을 열지 않는가
- 작가 클라이언트가 임의 작품을 건드릴 여지가 생긴다
- 태그, 이벤트, 가격 계산이 페이지마다 중복될 수 있다
- 권한 검사를 한 곳에 모으기 어렵다

따라서 이번 범위에서는 RLS를 넓히지 않고 RPC로 경계를 유지한다.

## 프론트 상태 설계

### 작품 업로드 JS
`upload-studio.js`는 `create` / `edit` 두 모드를 가진다.

상태 추가:
- `mode: 'create' | 'edit'`
- `editingNovelId`
- `existingCoverUrl`
- `loadedNovel`

주요 분기:
- `boot()`에서 쿼리 파라미터 확인
- 수정 모드면 기존 데이터 로드
- submit 시 `create_novel_with_episode` 대신 `update_novel_for_author` 호출

### 회차 업로드 JS
`episode-upload.js`는 `create` / `edit` 두 모드를 가진다.

상태 추가:
- `mode: 'create' | 'edit'`
- `editingEpisodeId`
- `loadedEpisode`
- `lockedNovelSlug`

주요 분기:
- 수정 모드면 대상 회차를 먼저 로드
- 작품 선택 UI 잠금
- submit 시 `create_episode_for_author_novel` 대신 `update_episode_for_author` 호출

### 크리에이터 대시보드 JS
`creator-dashboard.js`에 다음 상태를 추가한다.
- `salePanelOpen`
- `saleTargetNovelId`
- `saleForm`
- `activeSaleByNovelId`

주요 역할:
- 작품 목록을 가져올 때 할인 상태도 함께 표시
- `할인 등록` 클릭 시 패널 열기 및 기존 할인값 프리필
- 저장 시 `upsert_novel_sale_for_author` 호출
- 성공 후 대시보드 목록과 대표 카드 재렌더

## 문구 / UX 원칙

사용자는 코딩을 모른다는 전제를 유지한다. 따라서 화면 문구는 기술 용어보다 작업 결과가 바로 보이게 써야 한다.

예시:
- `RPC 오류` 같은 표현 금지
- `이 작품은 본인 계정으로 등록한 작품이 아니라 수정할 수 없습니다.` 처럼 바로 이해되는 문장 사용
- `할인율`, `시작일`, `종료일`, `편당 적용 가격`처럼 단순한 언어 사용

성공 메시지 예시:
- 작품 정보가 저장되었습니다.
- 회차 수정이 완료되었습니다.
- 할인 일정이 저장되었습니다.

실패 메시지 예시:
- 제목을 먼저 입력해주세요.
- 할인 종료일은 시작일보다 뒤여야 합니다.
- 이 작품은 수정 권한이 없습니다.

---

## 오류 처리

### 작품 수정
- 작품 ID가 없거나 잘못된 경우: 안내 후 대시보드로 복귀 링크 제공
- 권한 없는 작품인 경우: 읽기 전용 안내 카드 표시
- 표지 업로드 실패: 기존 표지 유지 여부를 명확히 안내

### 회차 수정
- 회차 ID가 없거나 잘못된 경우: 목록 복귀 유도
- 본문 로드 실패: 에디터 대신 오류 카드 표시
- 유료 회차인데 가격이 비어 있으면 저장 차단

### 할인 등록
- 종료일이 시작일보다 빠르면 저장 차단
- 이미 활성 할인 정보가 있으면 덮어쓴다는 안내 표시
- 저장 실패 시 할인 패널을 닫지 않고 값 유지

---

## 테스트 관점

### 수동 확인
- 대시보드에서 `작품 수정` 클릭 시 기존 값이 채워지는지
- 작품 수정 후 상세 페이지 제목/소개/태그가 바뀌는지
- 대시보드에서 `최근 회차 수정` 클릭 시 기존 회차가 열리는지
- 회차 수정 후 뷰어 본문이 바뀌는지
- 할인 등록 후 홈 세일 배너와 상세 가격 박스가 바뀌는지
- 권한 없는 계정으로 `?edit=` 접근 시 수정이 막히는지

### 자동 검증
- 스키마에 신규 RPC가 존재하는지 검사
- 업로드/회차 JS에 `edit` 모드 분기가 들어갔는지 검사
- 대시보드에 `작품 수정`, `할인 등록` 액션이 렌더되는지 검사
- 할인 RPC가 `novels`와 `events`/`event_items`를 함께 갱신하도록 설계되었는지 검사

## 구현 순서 제안

1. DB RPC 추가
2. 작품 수정 모드 추가
3. 회차 수정 모드 추가
4. 대시보드 액션 버튼 연결
5. 할인 등록 패널 추가
6. 할인 등록 후 홈/상세 반영 확인

이 순서가 좋은 이유는 권한 경계를 먼저 만들고, 그 위에 화면 기능을 얹을 수 있기 때문이다.

## 최종 결정

이번 작업은 **새로운 관리자 제품을 만드는 작업이 아니라, 기존 크리에이터 흐름을 실제 운영 가능한 수준으로 올리는 작업**으로 정의한다.

그래서 선택은 다음과 같다.
- 작품 수정: 기존 작품 업로드 화면 재사용
- 회차 수정: 기존 회차 업로드 화면 재사용
- 할인 등록: 대시보드 안의 단순 패널
- 데이터 쓰기: 직접 테이블 업데이트 대신 RPC

이 범위면 복잡도를 크게 늘리지 않으면서도, 작가가 작품을 등록한 뒤 다시 손보고 재노출시키는 핵심 운영 흐름을 완성할 수 있다.
